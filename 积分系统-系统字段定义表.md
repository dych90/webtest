# 积分系统-系统字段定义表
更新时间：2026-07-19  
状态：`定稿 V1.0 - 双轨制`

用途：把“等级系统 + 积分系统”的最新口径，冻结成可直接用于后端建模和接口设计的字段定义。

## 1. 定稿结论

本系统从 V1.0 起，采用双轨制：

1. `成长系统`
   - 用来体现长期成长
   - 负责等级、排行、皇冠、课时报告展示
   - 兑换礼物后不下降
2. `积分系统`
   - 用来兑换礼物
   - 可消耗
   - 商城兑换只扣积分，不扣成长等级

## 2. V1.0 固定业务规则

以下规则在 V1.0 视为冻结：

1. 本节课回课合格时：
   - `+5` 成长星
   - `+5` 积分
2. 本周练习奖励：
   - 老师一次给 `0 ~ 35` 成长星
   - 同时给 `0 ~ 35` 积分
   - 默认一比一同步
3. 成长升级规则：
   - `35` 成长星 = `1` 成长月亮
   - `10` 成长月亮 = `1` 成长太阳
4. 排名规则：
   - 排名只看 `累计成长星`
   - 不看可消费积分
5. 商城规则：
   - 普通礼物：只校验积分
   - 高级礼物：校验积分 + 成长等级门槛
6. V1.0 不启用积分过期
7. V1.0 不启用成长值过期

说明：

- 之前单轨制里的 `PointLot`、`PointSpendAllocation`、`moonBalance`、`sunBalance`，不再是 V1.0 必需结构
- 之前的“太阳卡作为商城兑换卡”口径，统一废弃；现在的 `sun` 只表示成长等级

## 3. 建模原则

V1.0 遵守下面 4 条原则：

1. 老师一次点击“已上课”，系统必须能完整结算一次奖励
2. 成长和积分要分账，但来源必须能对齐到同一次结算
3. 排名使用可复算的累计值，不使用会被消费影响的余额
4. 所有奖励、扣减、撤销都必须可追溯

## 4. V1.0 核心模型清单

以下 8 个对象为 V1.0 核心：

1. `LessonRewardSettlement`
2. `GrowthLedger`
3. `GrowthSummary`
4. `PointLedger`
5. `PointBalance`
6. `RewardCatalog`
7. `RewardRedemption`
8. `PointRuleConfig`

以下 2 个对象保留为阶段二能力，不阻塞 V1.0 核心结算：

1. `PracticePlan`
2. `PracticeCheckin`

## 6. 核心模型字段定稿

## 6.2 `GrowthLedger`

用途：记录成长值变动流水。  
V1.0 主要是正向增加；如果必须修正，用补发或冲销流水处理，而不是直接改总数。

| 字段 | 类型 | 必填 | 示例 | 说明 |
| --- | --- | --- | --- | --- |
| `_id` | `ObjectId` | 是 |  | 主键 |
| `studentId` | `ObjectId` | 是 |  | 学生 |
| `teacherId` | `ObjectId` | 是 |  | 老师 |
| `settlementId` | `ObjectId` | 否 |  | 对应结算单 |
| `changeAmount` | `Number` | 是 | `5` / `-5` | 成长星变动值 |
| `businessType` | `String` | 是 | `lesson_reward` | 业务类型 |
| `businessId` | `ObjectId` | 否 |  | 泛化业务主键 |
| `occurredAt` | `Date` | 是 |  | 发生时间 |
| `createdByType` | `String` | 是 | `teacher` | 操作人类型 |
| `createdById` | `ObjectId` | 否 |  | 操作人 |
| `remark` | `String` | 否 |  | 备注 |
| `createdAt` | `Date` | 是 |  | 创建时间 |

说明：

- `changeAmount` 使用有符号整数，便于后续冲销
- 但业务层默认只允许老师正向发放，负向变动需走作废或后台修正
- 负向冲销必须有明确 `remark`，用于解释成长值被扣回的原因

## 6.3 `GrowthSummary`

用途：保存学生成长汇总，供排行榜、学生列表、报表快速读取。

| 字段 | 类型 | 必填 | 示例 | 说明 |
| --- | --- | --- | --- | --- |
| `_id` | `ObjectId` | 是 |  | 主键 |
| `studentId` | `ObjectId` | 是 |  | 学生 |
| `teacherId` | `ObjectId` | 是 |  | 老师 |
| `totalGrowthStars` | `Number` | 是 | `428` | 累计成长星 |
| `totalLessonGrowthStars` | `Number` | 是 | `80` | 累计上课成长星 |
| `totalPracticeGrowthStars` | `Number` | 是 | `348` | 累计练习成长星 |
| `lastRewardAt` | `Date` | 否 |  | 最近奖励时间 |
| `updatedAt` | `Date` | 是 |  | 更新时间 |

索引建议：

- `studentId + teacherId` 唯一索引
- `teacherId + totalGrowthStars desc`

不入库、只在服务层推导的展示字段：

- `sunCount = floor(totalGrowthStars / 350)`
- `moonRemainder = floor((totalGrowthStars % 350) / 35)`
- `starRemainder = totalGrowthStars % 35`
- `rankScore = totalGrowthStars`
- `currentLevelType`

定稿决定：

- `currentLevelLabel` 不入库
- `sunCount`、`moonRemainder`、`starRemainder` 不入库
- 这些都由服务层实时推导

## 6.4 `PointLedger`

用途：记录积分增减流水。

| 字段 | 类型 | 必填 | 示例 | 说明 |
| --- | --- | --- | --- | --- |
| `_id` | `ObjectId` | 是 |  | 主键 |
| `studentId` | `ObjectId` | 是 |  | 学生 |
| `teacherId` | `ObjectId` | 是 |  | 老师 |
| `settlementId` | `ObjectId` | 否 |  | 对应结算单 |
| `changeAmount` | `Number` | 是 | `5` / `-120` | 积分变动值 |
| `businessType` | `String` | 是 | `reward_redeem` | 业务类型 |
| `businessId` | `ObjectId` | 否 |  | 泛化业务主键 |
| `occurredAt` | `Date` | 是 |  | 发生时间 |
| `createdByType` | `String` | 是 | `teacher` | 操作人类型 |
| `createdById` | `ObjectId` | 否 |  | 操作人 |
| `remark` | `String` | 否 |  | 备注 |
| `createdAt` | `Date` | 是 |  | 创建时间 |

说明：

- `changeAmount > 0` 表示发积分
- `changeAmount < 0` 表示扣积分
- 普通商城兑换不允许造成负积分

- 任何可能造成负积分的动作必须填写 `remark`，页面可通过欠账接口追溯原因

## 9. 接口载荷定稿

## 9.1 老师点击“已上课”结算奖励

`POST /lesson-records/:id/reward-settlement`

```json
{
  "issueLessonReward": true,
  "practiceRewardValue": 28,
  "remark": "本周练习较稳定，回课合格"
}
```

后端行为：

1. 创建 `LessonRewardSettlement`
2. 写入 `GrowthLedger`
3. 更新 `GrowthSummary`
4. 写入 `PointLedger`
5. 更新 `PointBalance`

## 9.2 学生管理页排行

`GET /api/reward-rankings`

默认返回成长值排行榜。学生管理页、皇冠展示必须使用成长值排行，不使用可消费积分排行。

```json
{
  "rankingMode": "growth",
  "crownMode": "growth",
  "list": [
    {
      "studentId": "s1",
      "name": "张三",
      "totalGrowthStars": 428,
      "sunCount": 1,
      "moonRemainder": 2,
      "starRemainder": 8,
      "availablePoints": 126,
      "rank": 1,
      "crownType": "gold"
    }
  ]
}
```

说明：

1. 皇冠只按成长值排行发放：第 1 名金冠，第 2 名银冠，第 3 名铜冠
2. 可消费积分可作为商城分析指标，但不作为学生管理页皇冠依据
3. 学生兑换商品后不会降低成长等级，也不会因为消费积分导致皇冠掉级

## 9.3 商城兑换

`POST /api/reward-redemptions`

```json
{
  "studentId": "s1",
  "rewardCode": "gift_headphone"
}
```

后端校验：

1. 商品是否上架
2. 当前积分是否足够
3. 当前成长等级是否达标
4. 扣积分成功后写兑换记录
5. 兑换不允许造成积分欠账

## 9.4 积分欠账查询

`GET /api/students/:id/point-debt`

当 `PointBalance.availablePoints < 0` 时返回欠账明细：

```json
{
  "hasDebt": true,
  "debtPoints": 12,
  "outstandingDebtEvents": [
    {
      "businessType": "lesson_reward",
      "reasonLabel": "上课奖励作废回收积分",
      "reasonDetail": "原课时记录录错",
      "debtIncrease": 12,
      "outstandingContribution": 12
    }
  ]
}
```

说明：

1. 欠账不是一个单独手填字段，而是由 `PointLedger` 流水推导
2. 欠账原因来自 `businessType + settlementId + remark`
3. 每一笔造成或加深欠账的流水都能追溯到业务来源

## 9.5 人工积分修正

`POST /api/students/:id/point-adjustments`

```json
{
  "changeAmount": -10,
  "reason": "历史积分迁移时多录入 10 分"
}
```

后端校验：

1. `changeAmount` 不能为 0
2. `reason` 必填
3. 正向修正只增加可消费积分，不增加成长等级
4. 负向修正允许出现欠账，但必须能在欠账接口看到原因

## 10. 对旧骨架的影响

以下旧方向现在明确不再继续：

1. `PointLot`
2. `PointSpendAllocation`
3. `star/moon/sun` 作为可消费余额
4. 星星自动过期
5. 月亮或太阳直接作为商城货币

## 11. 结论

双轨制定稿后的关键收口是：

1. `LessonRewardSettlement` 负责把老师一次操作固化成业务真相
2. `GrowthSummary.totalGrowthStars` 是排名唯一基线
3. `PointBalance.availablePoints` 是商城唯一消费基线
4. 所有等级展示都从成长总值推导，不再单独存冗余等级字段
